# EstateFlow Customer Success FTE ‚Äî Skills Manifest

Agent Skills are reusable capabilities the FTE can invoke at defined points in the processing pipeline. Each skill has a clear trigger condition, typed inputs, and typed outputs. Skills are independently testable and framework-agnostic ‚Äî they can be implemented as MCP tools, OpenAI function tools, or standalone Python functions.

---

## 1. Knowledge Retrieval Skill

- **When to use:** Customer asks product questions, needs how-to guidance, or reports an issue that may have a documented solution
- **Inputs:**
  - `query` (str) ‚Äî the customer's question or topic to search for
  - `max_results` (int, default: 3) ‚Äî maximum number of documentation sections to return
- **Outputs:**
  - `results` (list) ‚Äî matching documentation sections with relevance scores
  - `formatted_context` (str) ‚Äî results formatted as a readable block, ready to inject into the agent prompt
  - `result_count` (int) ‚Äî number of results returned

---

## 2. Sentiment Analysis Skill

- **When to use:** Every customer message ‚Äî run first, before any other skill
- **Inputs:**
  - `message` (str) ‚Äî the raw customer message text
- **Outputs:**
  - `sentiment` (str) ‚Äî one of: `positive`, `neutral`, `negative`, `frustrated`, `angry`
  - `confidence` (float, 0.0‚Äì1.0) ‚Äî confidence score for the classification
  - `signals_found` (list[str]) ‚Äî the specific words or phrases that triggered the classification
  - `is_negative` (bool) ‚Äî true if sentiment is negative, frustrated, or angry
  - `escalation_advised` (bool) ‚Äî true if sentiment alone warrants escalation
  - `tone_guidance` (str) ‚Äî instructions for how the agent should adjust response tone

---

## 3. Escalation Decision Skill

- **When to use:** After sentiment is known and before generating a response; also checks sentiment trend from session memory
- **Inputs:**
  - `conversation_context` (str) ‚Äî full session history including prior turns and channel switches
  - `sentiment_trend` (list[str]) ‚Äî list of sentiment values from all prior turns in this session
  - `message` (str) ‚Äî the current customer message
  - `plan` (str) ‚Äî the customer's current plan (starter / professional / team / brokerage)
  - `troubleshooting_attempts` (int) ‚Äî number of prior troubleshooting turns in this session
- **Outputs:**
  - `should_escalate` (bool) ‚Äî whether to escalate this interaction
  - `reason` (str) ‚Äî human-readable explanation of why escalation was triggered
  - `level` (str) ‚Äî escalation level: L1 (next day) / L2 (4 hrs) / L3 (1 hr) / L4 (immediate)
  - `rule_triggered` (str) ‚Äî which escalation rule fired (e.g. "Rule 4 ‚Äî Billing Dispute")
  - `sentiment_worsening` (bool) ‚Äî true if the last 2 sentiments are both negative/frustrated/angry

---

## 4. Channel Adaptation Skill

- **When to use:** Before sending any response ‚Äî always the final step before delivery
- **Inputs:**
  - `response_text` (str) ‚Äî the raw response generated by the agent
  - `target_channel` (str) ‚Äî the channel to format for: `email`, `whatsapp`, or `web_form`
  - `customer_name` (str, optional) ‚Äî customer's first name for greeting personalisation
- **Outputs:**
  - `formatted_response` (str) ‚Äî channel-formatted response ready to send
  - `word_count` (int) ‚Äî word count of the formatted response
  - `truncated` (bool) ‚Äî true if the response was shortened to fit channel limits

**Channel rules applied:**

| Channel | Tone | Max Length | Formatting | Greeting | Sign-off |
|---|---|---|---|---|---|
| Email | Formal, structured | 500 words | Markdown supported | "Hi [Name]," | Signature + email |
| WhatsApp | Conversational, concise | 80 words / 300 chars | Plain text only | None | None |
| Web Form | Semi-formal | 300 words | Markdown supported | "Hi [Name]," | Closing line only |

## 5. Customer Identification Skill

- **When to use:** On every incoming message ‚Äî the very first skill to run before anything else
- **Inputs:**
  - `email` (str, optional) ‚Äî customer email address ‚Äî primary identifier
  - `phone` (str, optional) ‚Äî customer phone number ‚Äî secondary identifier (WhatsApp)
  - `name` (str, optional) ‚Äî customer display name
  - `plan` (str, optional) ‚Äî customer's current plan
  - `channel` (str) ‚Äî the channel this message arrived from
- **Outputs:**
  - `customer_id` (str) ‚Äî unified customer ID used across all tools and channels
  - `merged_history` (list) ‚Äî all prior tickets and sessions for this customer across every channel
  - `is_new_customer` (bool) ‚Äî true if no prior interactions found
  - `channels_used` (list[str]) ‚Äî all channels this customer has previously contacted via
  - `cross_channel_match` (bool) ‚Äî true if email and phone were successfully linked to the same record

**Identity resolution rules:**
1. Email is always the primary key
2. Phone is the secondary key (used when email is unavailable, e.g. WhatsApp-only customers)
3. If both email and phone are provided and match different records, merge under the email record
4. At least one of email or phone must be provided

## Additional Skills (Discovered During Incubation)

The following two skills were not in the original spec but were discovered as necessary during prototype testing.

### 6. Multi-Question Parser Skill

- **When to use:** On every email message before knowledge base search ‚Äî email messages frequently contain 2‚Äì3 questions in one message
- **Inputs:**
  - `message` (str) ‚Äî the raw customer message
  - `channel` (str) ‚Äî the source channel (only applies meaningful parsing on `email`)
- **Outputs:**
  - `questions` (list[str]) ‚Äî list of individual questions extracted
  - `question_count` (int) ‚Äî number of distinct questions found
  - `has_multiple_questions` (bool) ‚Äî true if more than one question was found

**Why this was needed:** Analysis of 19 email tickets showed that multi-part questions are common among Professional and Team plan customers. Answering only the first question leaves the customer feeling unheard and generates unnecessary follow-up tickets.

---

### 7. Retention Trigger Skill

- **When to use:** When cancellation language is detected in any message ‚Äî runs before the escalation decision to give retention one attempt first
- **Inputs:**
  - `message` (str) ‚Äî the raw customer message
- **Outputs:**
  - `cancellation_detected` (bool) ‚Äî true if cancellation language was found
  - `signal_phrase` (str) ‚Äî the exact phrase that triggered detection
  - `retention_question` (str) ‚Äî the single question the agent should ask before proceeding
  - `escalate_if_confirmed` (bool) ‚Äî always true; if customer confirms intent after engagement, escalate per Rule 3

**Why this was needed:** TKT-050 in the sample set showed a customer who opened with "before I cancel..." and re-engaged when asked one good question. Without this skill, the agent would jump straight to escalation and lose the retention window.
